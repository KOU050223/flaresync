# flaresync — 設計判断の記録

## なぜ setInterval ではなく alarm() を使うのか

**判断:** ティック配信には `alarm()` APIを使う

**理由:**
DOは `ctx.acceptWebSocket()` で接続を受けると、接続中でもHibernation（冬眠）できる。
Hibernation中は `setInterval` が完全に停止するため、「パッチが届かない」バグが本番で発生する。
`alarm()` はストレージに永続化されるため、Hibernation中でも確実に発火する。

**実装方針:**
- `set` トラップで変更検知 → alarmが未設定なら `setAlarm(now + 50ms)`
- `alarm()` で差分配信 → 接続者がいれば次のalarmを再スケジュール
- 接続者ゼロで自動停止（無駄なコスト防止）

## なぜ Proxy を使うのか

**判断:** 状態変更の検知には `Proxy` の `set` トラップを使う

**理由:**
ユーザーに `setState()` のような専用メソッドを強制せず、普通のJavaScriptの代入（`state.hp = 90`）で同期が動く体験を提供したい。

**注意点:**
ネストしたオブジェクトを再帰Proxy化する際、`get` トラップのたびに新しいProxyを生成すると参照等価性が壊れる（`obj.a === obj.a` が `false` になる）。
→ `WeakMap` キャッシュで同じパスには同じProxyを返す（TODO 042）

## なぜ差分（dirty）を溜めてまとめて送るのか

**判断:** 変更のたびに即送信せず、50msごとにまとめて送る

**理由:**
ARゲームでは1ティック内に `x`, `y`, `hp` が全部変わることがある。
変更のたびに送ると3回のWebSocketメッセージになるが、まとめると1回で済む。
通信回数が減ることでCloudflareの課金も抑えられる。

---

## なぜ Cloudflare 特化なのか

**判断:** Node.js等で動く汎用ライブラリにしない

**理由:**
汎用にすると `getWebSockets()` や `alarm()` などCloudflare固有APIを使えなくなり、設定が増える。
「Cloudflareだけで動く代わりに設定ゼロ」という尖った体験を提供することが差別化になる。

---

## ターゲットをゲーム/リアルタイムに絞る理由

**判断:** 共同編集（CRDT）やIoTは最初のスコープ外

**理由:**
ゲームと共同編集では「競合解消」の要件が真逆。
- ゲーム: Last Write Wins（最後に書いた値が勝つ）で十分
- 共同編集: CRDT（数学的な自動マージ）が必要

CRDTを入れると設計が複雑になるため、まずLWWで動かして後から拡張を検討する。
